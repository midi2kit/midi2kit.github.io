MIDI2Kit が macOS 15+ API に対応: UMP エンドポイント、MIDI-CI v1.2、Network MIDI 2.0

macOS 15 (Sequoia) で導入された新しい CoreMIDI API との統合が完了しました。macOS 26.4 (Tahoe) での検証結果も含め、Apple の最新 MIDI 2.0 サブシステムとの橋渡しを実現します。

はじめに

MIDI2Kit は、macOS 15 (Sequoia) で新たに公開された CoreMIDI API への統合を完了しました。MIDIUMPEndpointManager によるネイティブな UMP エンドポイント検出、MIDICIDeviceManager によるシステム管理の MIDI-CI セッション連携、そして MIDI-CI v1.2 の仕様準拠 — これらすべてが今回のアップデートで実現しています。

さらに、macOS 26.4 (Tahoe) ベータ環境での実機検証も行いました。新しい Network MIDI 2.0 (UDP) の受信パス対応と、MIDIUMPMutableEndpoint の改善を確認しています。検証で得られた知見についても本記事で詳しく解説します。

今回の変更はすべて @available(macOS 15, *) ガード付きで実装されています。macOS 14 以前の環境でも、デフォルトの動作は一切変わりません。

macOS 15+ CoreMIDI の新機能

macOS 15 (Sequoia) は、CoreMIDI に対して MIDI 2.0 時代に向けた重要な API 拡張をもたらしました。これらは単なる便利メソッドの追加ではなく、MIDI 2.0 エコシステムとの深い統合を可能にする基盤となるものです。

  • MIDIUMPEndpointManager — UMP エンドポイントをシステムレベルで管理するクラス。接続されたデバイスが UMP をネイティブサポートしているかどうかを OS が直接把握し、アプリに通知します。
  • MIDICIDeviceManager — システムが管理する MIDI-CI セッションの情報を提供します。Logic Pro や GarageBand などの Apple 製アプリが内部的に使用している CI セッションを、サードパーティアプリからも参照できます。
  • MIDISendEventList — MIDI 2.0 UMP パケットをネイティブに送信する新しい API。従来の MIDISend に代わる、UMP ファーストな送信インターフェースです。
  • Network MIDI 2.0 (UDP) — macOS 26.4 で追加。RTP-MIDI の後継として、UDP 上で UMP パケットを直接送受信します。
  • MIDIUMPMutableEndpoint — macOS 26.4 で改善。仮想 UMP エンドポイントの作成と Function Block 登録がより柔軟になりました。

MIDI2Kit の新機能

1. UMP エンドポイント検出 (#36)

従来、MIDI ソース・デスティネーションが UMP をネイティブサポートしているかどうかを判定するには、デバイス名のパターンマッチングや独自のヒューリスティックに頼るしかありませんでした。macOS 15 の MIDIUMPEndpointManager を活用することで、OS から直接この情報を取得できるようになりました。

MIDISourceInfo と MIDIDestinationInfo に新しい umpEndpointInfo フィールドが追加されました。このフィールドには、デバイスがサポートする UMP バージョン、Function Block の数、プロトコルネゴシエーションの結果などが含まれます。また、簡便な判定のために isUMPNative プロパティも追加しています。

import MIDI2Kit

let client = try MIDI2Client(name: "MyApp")
try await client.start()

for await event in await client.makeEventStream() {
    if case .deviceDiscovered(let device) = event {
        // UMP ネイティブデバイスかどうかを確認
        let sourceInfo = await client.sourceInfo(for: device)
        if sourceInfo?.isUMPNative == true {
            print("\(device.displayName) は UMP ネイティブデバイスです")
            print("UMP バージョン: \(sourceInfo?.umpEndpointInfo?.umpVersion ?? "不明")")
        }
    }
}

UMP エンドポイントの変更通知は、既存の setupChanged ストリームに統合されています。MIDIUMPEndpointManager の通知と CoreMIDI のセットアップ変更通知が単一のストリームに集約されるため、アプリ側で複数のオブザーバーを管理する必要はありません。

2. スマート SysEx 送信 (#38)

UMP ネイティブデバイスへの SysEx 送信には、従来の MIDISend ベースのレガシーパスよりも、MIDISendEventList を使った UMP ネイティブパスの方が信頼性が高いケースがあります。しかし KORG Module Pro のような BLE MIDI デバイスは、UMP 送信に対応していないため、レガシーパスを維持する必要があります。

この問題を解決するために、SysExSendStrategy を導入しました。

// デフォルト: レガシーパスのみ (KORG BLE MIDI などに安全)
config.sysExSendStrategy = .legacyOnly

// 自動選択: UMP ネイティブデバイスには MIDISendEventList を使用
config.sysExSendStrategy = .auto

// UMP のみ: UMP ネイティブデバイス専用アプリに
config.sysExSendStrategy = .umpOnly

.auto モードでは、送信先デバイスの isUMPNative を実行時に確認し、適切なパスを自動選択します。既存の設定を変更せずに最新 API の恩恵を受けたい場合は、.auto への移行を推奨します。

3. MIDICIDeviceManager 統合 (#37)

macOS に付属する Logic Pro、GarageBand、Audio MIDI Setup などの Apple 製アプリは、内部的に MIDI-CI セッションを確立しています。これらのアプリが使用している MUID が MIDI2Kit の Discovery で検出されると、実際には応答のないデバイスとして登録されてしまい、Property Exchange のタイムアウトを引き起こす問題がありました。

MIDICIDeviceManager との統合により、Apple 内蔵 CI セッションの MUID を自動的にブラックリストに追加するようになりました。

// 以前: 手動でブラックリストを管理する必要があった
var ciConfig = CIManagerConfiguration()
ciConfig.muidBlacklist = [0x0012A4B3, 0x0056C2D1] // 毎回異なる値

// 現在: MIDICIDeviceManager が自動的に処理
// 追加の設定は不要です

ブラックリストは MIDICIDeviceManager の通知に基づいて動的に更新されます。Apple 製アプリの起動・終了に伴う CI セッションの変化にも追従するため、手動での muidBlacklist 設定は不要になりました。

4. MIDI-CI v1.2 準拠 (#41)

MIDI-CI の仕様は v1.1 から v1.2 にアップデートされ、いくつかの重要な変更が加えられました。MIDI2Kit はすべてのメッセージビルダーを v1.2 対応に更新しています。

最も重要な変更点は、PE Notify メッセージタイプの変更です。v1.1 では Notify に 0x38 を使用していましたが、v1.2 では 0x3F に変更されました。MIDI2Kit は接続相手の ciVersion を自動的に確認し、適切なメッセージタイプを使い分けます。

// DiscoveredDevice に ciVersion フィールドが追加されました
for await event in await client.makeEventStream() {
    if case .deviceDiscovered(let device) = event {
        // デバイスがサポートする CI バージョンを確認
        print("CI バージョン: \(device.ciVersion)")
        // => "1.1" または "1.2"
    }
}

// メッセージビルダーに ciVersion パラメータが追加されました
// (デフォルトは v1.1 で後方互換性を維持)
let ackMessage = CIMessageBuilder.buildACK(
    sourceMUID: myMUID,
    destinationMUID: targetMUID,
    originalTransactionSubID: 0x70,
    ciVersion: device.ciVersion  // デバイスのバージョンに合わせる
)

また、これまで MIDI-CI 仕様では定義されていながら MIDI2Kit に実装がなかった ACK/NAK ビルダーが追加されました。Responder 実装でより細かな応答制御が可能になります。

5. Profile Configuration (#42)

MIDI-CI の Profile Configuration メッセージに対応しました。Profile は、特定の機能セット（Piano、Drawbar Organ、Guitar など）を標準化された方法でネゴシエートするための仕組みです。

// Profile Inquiry の送信
let inquiryBytes = CIMessageBuilder.buildProfileInquiry(
    sourceMUID: myMUID,
    destinationMUID: targetMUID
)
try await transport.send(inquiryBytes, to: destination)

// Profile の有効化要求
let setOnBytes = CIMessageBuilder.buildProfileSetOn(
    sourceMUID: myMUID,
    destinationMUID: targetMUID,
    profile: MIDICIProfileID(standard: .piano),
    channel: 0,
    numChannels: 1  // CI v1.2 の numChannels フィールド
)
try await transport.send(setOnBytes, to: destination)

今回追加されたビルダーは以下の通りです:

  • buildProfileInquiry — デバイスがサポートするプロファイルを問い合わせます
  • buildProfileReply — サポートするプロファイルのリストで応答します
  • buildProfileSetOn / buildProfileSetOff — プロファイルの有効化・無効化を要求します
  • buildProfileEnabledReport / buildProfileDisabledReport — プロファイルの状態変化を通知します

パーサー側も CIMessageParser に parseProfileReply と parseSetProfile が追加されました。

macOS 26.4 (Tahoe) での検証結果

macOS 26.4 ベータでの実機検証で得られた知見を共有します。これらはベータ段階の結果であり、正式リリースまでに変更される可能性があります。

registerFunctionBlocks: エラー解消

macOS 15 では、MIDIUMPMutableEndpoint に Function Block を登録しようとすると全件エラー (OSStatus -10849) が発生していました。macOS 26.4 ではこのエラーが解消され、Function Block の登録が正常に動作することを確認しました。

ただし、動作する構成は Baseline のみです。16 パターンのパラメータスイープ (グループ数 1/2/4、チャンネル割り当て MIDI 1.0/MIDI 2.0、Block Type 入力/出力/双方向、UMP バージョン 1.0/1.1) を試した結果、安定して認識されるのは最もシンプルな Baseline 構成 (グループ 1、MIDI 2.0 チャンネル、双方向、UMP 1.1) のみでした。他の構成では、登録は成功するものの Audio MIDI Setup での可視性に問題が残っています。

Network MIDI 2.0 (UDP)

macOS 26.4 で新たに追加された Network MIDI 2.0 は、UDP 上で UMP パケットを直接送受信する仕組みです。MIDI2Kit では受信パスの対応を完了しています。

注意点として、Network MIDI 2.0 デバイスが接続されると、CoreMIDITransport が既存の BLE MIDI セッションを Network MIDI 2.0 セッションとして誤判定するケースが確認されています。この問題は次のリリースで修正予定です。

MIDIDriverKit について

macOS 26.4 の MIDIDriverKit は、カーネル拡張なしで MIDI ドライバーを実装するための新しい仕組みです。調査の結果、Swift Package Manager ベースのライブラリでは利用できない DriverKit 専用の entitlement が必要なため、MIDI2Kit では追求しないことを決定しました。

コード例: SysExSendStrategy の実際の使い方

以下は、UMP ネイティブデバイスと MIDI 1.0 デバイスが混在する環境でのセットアップ例です。

import MIDI2Kit

// 自動戦略で MIDI2Client を設定
var config = MIDI2ClientConfiguration()
config.sysExSendStrategy = .auto  // UMP ネイティブデバイスを自動検出

// KORG Module Pro など BLE MIDI デバイスが混在する場合
// config.sysExSendStrategy = .legacyOnly  // デフォルト (安全)

let client = try MIDI2Client(name: "MyApp", configuration: config)
try await client.start()

// デバイス検出後の Property Exchange は通常通り動作
// 送信戦略は内部で自動的に適用されます
for await event in await client.makeEventStream() {
    if case .deviceDiscovered(let device) = event {
        // UMP ネイティブデバイスには MIDISendEventList を使用
        // MIDI 1.0 デバイスには従来の MIDISend を使用
        // アプリ側のコードは変更不要
        let info = try await client.getDeviceInfo(from: device)
        print("デバイス情報: \(info)")
    }
}

後方互換性

今回の変更はすべて後方互換性を維持しています。

  • @available(macOS 15, *) ガード — 新しい CoreMIDI API はすべて可用性チェックで保護されています。macOS 14 以前では既存のコードパスが使用されます。
  • デフォルト動作は変更なし — SysExSendStrategy のデフォルトは .legacyOnly、ciVersion のデフォルトは v1.1 です。明示的にオプトインするまで、既存のアプリの動作は変わりません。
  • 705 テスト全パス — 既存のすべてのテストが引き続き通過することを確認しています。新機能のテストも含めて 705 件のテストが 100% パスしています。

macOS 15 未満をサポートする必要があるプロジェクトでも、MIDI2Kit を安全にアップデートできます。新機能を利用したい場合は、if #available(macOS 15, *) で囲んだコードブロック内で新しい API を呼び出してください。

MIDI2Kit を始める

macOS 15+ API 対応を含む最新版は GitHub で公開しています。Swift Package Manager で簡単に導入できます。
