MIDI2Kit が macOS 15+ API に対応: UMP エンドポイント、MIDI-CI v1.2、Network MIDI 2.0

macOS 15 (Sequoia) の SDK で確認できる CoreMIDI クラスとの統合が完了しました。macOS 26.4 (Tahoe) での検証結果も含め、Apple の MIDI 2.0 サブシステムとの橋渡しを実現します。

はじめに

MIDI2Kit は、macOS 15 (Sequoia) の SDK ヘッダーで確認できる CoreMIDI クラスとの統合を完了しました。MIDIUMPEndpointManager によるネイティブな UMP エンドポイント検出、MIDICIDeviceManager によるシステム管理の MIDI-CI セッション連携、そして MIDI-CI v1.2 の仕様準拠 — これらすべてが今回のアップデートで実現しています。なお、これらのクラスは Apple のリリースノートでは特に言及されていませんが、SDK に含まれ実行時に利用可能です。

macOS 26.4 (Tahoe) の公式リリースノートで確認できる CoreMIDI の変更は Network MIDI 2.0 (UDP) のサポート追加のみです。加えて、実機検証により MIDIUMPMutableEndpoint の改善も確認しています。

今回の変更はすべて @available(macOS 15, *) ガード付きで実装されています。macOS 14 以前の環境でも、デフォルトの動作は一切変わりません。

本リリースで使用する CoreMIDI API

MIDI2Kit が統合した CoreMIDI API と、それぞれの出自を整理します。Apple のリリースノートで言及されていない API も含まれますが、SDK ヘッダーに存在し実行時に利用可能です。

  • MIDIUMPEndpointManager (macOS 15.0+) — UMP エンドポイントをシステムレベルで管理するクラス。macOS 15 SDK ヘッダーで確認。リリースノートでの言及なし。
  • MIDICIDeviceManager (macOS 15.0+) — CoreMIDI が認識している CI 対応デバイスのリストを提供。実機テストでは Apple の SysEx ID (0x11) を持つ CI 参加者が含まれることを確認。macOS 15 SDK ヘッダーで確認。リリースノートでの言及なし。
  • MIDISendEventList (macOS 11.0+) — MIDI 2.0 UMP パケットのネイティブ送信 API。macOS 15 の新 API ではなく、macOS Big Sur から存在。MIDI2Kit のスマート SysEx ルーティングの基盤として活用。
  • Network MIDI 2.0 (UDP) (macOS 26.4) — macOS 26.4 の公式リリースノートで確認できる唯一の CoreMIDI 関連変更。UDP 上で UMP パケットを直接送受信。
  • MIDIUMPMutableEndpoint (macOS 15.0+) — macOS 26.4 での実機テストにより、registerFunctionBlocks エラーの解消を確認（ただしパラメータサポートは限定的）。

MIDI2Kit の新機能

1. UMP エンドポイント検出 (#36)

従来、MIDI ソース・デスティネーションが UMP をネイティブサポートしているかどうかを判定するには、デバイス名のパターンマッチングや独自のヒューリスティックに頼るしかありませんでした。macOS 15 の MIDIUMPEndpointManager を活用することで、OS から直接この情報を取得できるようになりました。

MIDISourceInfo と MIDIDestinationInfo に新しい umpEndpointInfo フィールドが追加されました。このフィールドには、デバイスがサポートする UMP バージョン、Function Block の数、プロトコルネゴシエーションの結果などが含まれます。また、簡便な判定のために isUMPNative プロパティも追加しています。

import MIDI2Kit

let client = try MIDI2Client(name: "MyApp")
try await client.start()

for await event in await client.makeEventStream() {
    if case .deviceDiscovered(let device) = event {
        // UMP ネイティブデバイスかどうかを確認
        let sourceInfo = await client.sourceInfo(for: device)
        if sourceInfo?.isUMPNative == true {
            print("\(device.displayName) は UMP ネイティブデバイスです")
            print("UMP バージョン: \(sourceInfo?.umpEndpointInfo?.umpVersion ?? "不明")")
        }
    }
}

UMP エンドポイントの変更通知は、既存の setupChanged ストリームに統合されています。MIDIUMPEndpointManager の通知と CoreMIDI のセットアップ変更通知が単一のストリームに集約されるため、アプリ側で複数のオブザーバーを管理する必要はありません。

2. スマート SysEx 送信 (#38)

UMP ネイティブデバイスへの SysEx 送信には、従来の MIDISend ベースのレガシーパスよりも、MIDISendEventList を使った UMP ネイティブパスの方が信頼性が高いケースがあります。しかし KORG Module Pro のような BLE MIDI デバイスは、UMP 送信に対応していないため、レガシーパスを維持する必要があります。

この問題を解決するために、SysExSendStrategy を導入しました。

// デフォルト: レガシーパスのみ (KORG BLE MIDI などに安全)
config.sysExSendStrategy = .legacyOnly

// 自動選択: UMP ネイティブデバイスには MIDISendEventList を使用
config.sysExSendStrategy = .auto

// UMP のみ: UMP ネイティブデバイス専用アプリに
config.sysExSendStrategy = .umpOnly

.auto モードでは、送信先デバイスの isUMPNative を実行時に確認し、適切なパスを自動選択します。既存の設定を変更せずに最新 API の恩恵を受けたい場合は、.auto への移行を推奨します。

3. MIDICIDeviceManager 統合 (#37)

MIDI-CI Discovery を行うと、Apple の manufacturer SysEx ID (0x11) を持つ CI 参加者が検出されることがあります。これらの参加者は Property Exchange リクエストに応答しないため、タイムアウトの原因となっていました。これらの正体は Apple の公式ドキュメントやリリースノートには記載されておらず、実機テストで経験的に確認したものです。

MIDI2Kit は macOS 15 SDK の MIDICIDeviceManager クラスを使い、起動時に discoveredCIDevices を参照して、Apple の SysEx ID (0x11 0x00 0x00) を持つデバイスの MUID をヒューリスティックにブラックリストへ追加します。これは公式ガイダンスに基づくものではなく、経験的な回避策です。

// 以前: 手動でブラックリストを管理する必要があった
var ciConfig = CIManagerConfiguration()
ciConfig.muidBlacklist = [0x0012A4B3, 0x0056C2D1] // 毎回異なる値

// 現在: MIDICIDeviceManager が自動的に処理
// 追加の設定は不要です

ブラックリストは MIDICIDeviceManager の通知に基づいて動的に更新されます。macOS 15+ では手動での muidBlacklist 設定は不要になりますが、手動 API はフォールバックとして引き続き利用可能です。

4. MIDI-CI v1.2 準拠 (#41)

MIDI-CI v1.2 仕様 (M2-101-UM, MIDI Association が 2023年6月に公開) では、Management Messages (ACK/NAK)、Notify メッセージタイプの変更、バージョンネゴシエーションが追加されました。これは MIDI Association の仕様であり、Apple の macOS 機能ではありません。MIDI2Kit のメッセージビルダー/パーサーをこの仕様に準拠させました。

最も重要な変更点は、PE Notify メッセージタイプの変更です。v1.1 では Notify に 0x38 を使用していましたが、v1.2 では 0x3F に変更されました。MIDI2Kit は接続相手の ciVersion を自動的に確認し、適切なメッセージタイプを使い分けます。

// DiscoveredDevice に ciVersion フィールドが追加されました
for await event in await client.makeEventStream() {
    if case .deviceDiscovered(let device) = event {
        // デバイスがサポートする CI バージョンを確認
        print("CI バージョン: \(device.ciVersion)")
        // => "1.1" または "1.2"
    }
}

// メッセージビルダーに ciVersion パラメータが追加されました
// (デフォルトは v1.1 で後方互換性を維持)
let ackMessage = CIMessageBuilder.buildACK(
    sourceMUID: myMUID,
    destinationMUID: targetMUID,
    originalTransactionSubID: 0x70,
    ciVersion: device.ciVersion  // デバイスのバージョンに合わせる
)

また、これまで MIDI-CI 仕様では定義されていながら MIDI2Kit に実装がなかった ACK/NAK ビルダーが追加されました。Responder 実装でより細かな応答制御が可能になります。

5. Profile Configuration (#42)

MIDI-CI の Profile Configuration メッセージに対応しました。Profile は、特定の機能セット（Piano、Drawbar Organ、Guitar など）を標準化された方法でネゴシエートするための仕組みです。

// Profile Inquiry の送信
let inquiryBytes = CIMessageBuilder.buildProfileInquiry(
    sourceMUID: myMUID,
    destinationMUID: targetMUID
)
try await transport.send(inquiryBytes, to: destination)

// Profile の有効化要求
let setOnBytes = CIMessageBuilder.buildProfileSetOn(
    sourceMUID: myMUID,
    destinationMUID: targetMUID,
    profile: MIDICIProfileID(standard: .piano),
    channel: 0,
    numChannels: 1  // CI v1.2 の numChannels フィールド
)
try await transport.send(setOnBytes, to: destination)

今回追加されたビルダーは以下の通りです:

  • buildProfileInquiry — デバイスがサポートするプロファイルを問い合わせます
  • buildProfileReply — サポートするプロファイルのリストで応答します
  • buildProfileSetOn / buildProfileSetOff — プロファイルの有効化・無効化を要求します
  • buildProfileEnabledReport / buildProfileDisabledReport — プロファイルの状態変化を通知します

パーサー側も CIMessageParser に parseProfileReply と parseSetProfile が追加されました。

macOS 26.4 (Tahoe) 詳細検証結果

UMPEndpointPoC テストハーネス (v3, Parameter Sweep) を macOS 26.4 (Build 25E246) で実行しました。外部 MIDI デバイスは未接続の状態での結果です。

Test A: registerFunctionBlocks — 16パターンスイープ

macOS 15 では registerFunctionBlocks が全 16 パターンで Foundation._GenericObjCError code=0 を送出していました。macOS 26.4 ではこのエラーが解消: 登録エラー 0/16、有効化エラー 0/16。

ただし、MIDIUMPEndpointManager で Function Block が可視になるのは 16 パターン中 1 パターンのみ:

パラメータ 動作する値
direction.bidirectional
maxSysEx8Streams0
midi1Info.notMIDI1
uiHint.unknown
markAsStatictrue
登録タイミングregister を enable の前に実行

残り 15 パターンは、Baseline から 1 つのパラメータを変更した構成です。エラーは返らないが、MIDIUMPEndpointManager で functionBlocks.count が 0 になります。以下のいずれかを変更すると FB が不可視に:

  • direction を .input または .output に変更
  • maxSysEx8Streams を 1 に変更
  • midi1Info を .unrestricted または .restrictBW に変更
  • uiHint を .receiver / .sender / .sndRcv に変更
  • markAsStatic を false に変更
  • 登録タイミングを setEnabled(true) の後に変更

macOS 15 では全 16 パターンが direction=0 (Unknown) でしたが、macOS 26.4 では Case #0 が direction=3 (bidirectional) を正しく報告しています。

Test B: MIDIInputPort UMP 受信

MIDIInputPortCreateWithProtocol(._2_0) で作成した入力ポートを全ソースに接続し 10 秒間監視。結果: 受信 UMP 0 パケット、Data 64 (SysEx7) 0 パケット。外部デバイス未接続のため、CoreMIDI が UMP パスで CI SysEx をフィルタリングしているかは検証できていません。

Test C: MIDIUMPEndpointManager 列挙

UMP エンドポイントは 1 件のみ（PoC 自身の仮想エンドポイント）。外部デバイス未接続のため、KORG KeyStage 等が UMP エンドポイントとして登録されるかは未確認。

Network MIDI 2.0 (UDP)

macOS 26.4 の公式リリースノートで Network MIDI 2.0 サポートが確認されています。MIDI2Kit の受信パスは MIDIInputPortCreateWithProtocol(._2_0) を使用しており、Network MIDI 2.0 の UMP ストリームを透過的に処理できるはずですが、Network MIDI 2.0 ピアが利用できなかったためテスト未実施。送信パスは現在 MIDISend (レガシー MIDI 1.0 API) を使用中。SysExSendStrategy.auto で UMP ネイティブデバイスには MIDISendEventList にルーティングしますが、Network MIDI 2.0 セッション経由のテストは未実施です。

MIDIDriverKit

仮想 MIDI 2.0 デバイスの代替手段として調査。結論: SPM パッケージには不適。C++ 必須、Xcode プロジェクト必須、DriverKit エンタイトルメント（Apple 承認必要）、SIP 有効環境必須。既存の MIDIDestinationCreateWithBlock / MIDIUMPMutableEndpoint アプローチが MIDI2Kit に適しています。

総合判定: PARTIAL-GO

MIDIUMPMutableEndpoint は macOS 26.4 で Baseline 構成で動作可能。FunctionBlock の全パラメータカスタマイズと UMP パスでの Data 64 受信は未検証。外部デバイス接続テストで広範な互換性が確認されるまで、レガシー API の継続使用を推奨します。

コード例: SysExSendStrategy の実際の使い方

以下は、UMP ネイティブデバイスと MIDI 1.0 デバイスが混在する環境でのセットアップ例です。

import MIDI2Kit

// 自動戦略で MIDI2Client を設定
var config = MIDI2ClientConfiguration()
config.sysExSendStrategy = .auto  // UMP ネイティブデバイスを自動検出

// KORG Module Pro など BLE MIDI デバイスが混在する場合
// config.sysExSendStrategy = .legacyOnly  // デフォルト (安全)

let client = try MIDI2Client(name: "MyApp", configuration: config)
try await client.start()

// デバイス検出後の Property Exchange は通常通り動作
// 送信戦略は内部で自動的に適用されます
for await event in await client.makeEventStream() {
    if case .deviceDiscovered(let device) = event {
        // UMP ネイティブデバイスには MIDISendEventList を使用
        // MIDI 1.0 デバイスには従来の MIDISend を使用
        // アプリ側のコードは変更不要
        let info = try await client.getDeviceInfo(from: device)
        print("デバイス情報: \(info)")
    }
}

後方互換性

今回の変更はすべて後方互換性を維持しています。

  • @available(macOS 15, *) ガード — 新しい CoreMIDI API はすべて可用性チェックで保護されています。macOS 14 以前では既存のコードパスが使用されます。
  • デフォルト動作は変更なし — SysExSendStrategy のデフォルトは .legacyOnly、ciVersion のデフォルトは v1.1 です。明示的にオプトインするまで、既存のアプリの動作は変わりません。
  • 705 テスト全パス — 既存のすべてのテストが引き続き通過することを確認しています。新機能のテストも含めて 705 件のテストが 100% パスしています。

macOS 15 未満をサポートする必要があるプロジェクトでも、MIDI2Kit を安全にアップデートできます。新機能を利用したい場合は、if #available(macOS 15, *) で囲んだコードブロック内で新しい API を呼び出してください。
